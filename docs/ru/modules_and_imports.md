# Модули и импорты

В этом документе описано, как **импортировать** и **использовать модули** в DataCode: синтаксис, типы модулей (файл и пакет), составные имена, базовый путь и встроенные модули. Детали реализации (байткод, ModuleObject, перемаппинг) см. в [Ядро / Internals — Система импорта модулей](./internals/module_import_system.md).

---

## Синтаксис импорта

### Импорт модуля (объект модуля)

```datacode
import ml
import plot
import settings_env
```

Модуль загружается, и его **объект пространства имён** привязывается к указанному имени. Обращение к экспортам — через имя модуля:

```datacode
import ml
let x = ml.load_mnist()
```

### Импорт отдельных имён из модуля

```datacode
from ml import load_mnist, NeuralNetwork
from settings_env import load_env, Settings
from core.config import get_settings, load_settings
```

- **Именованный**: `from M import X, Y` — импортирует из модуля `M` имена `X` и `Y` в текущую область видимости.
- **С псевдонимом**: `from M import X as Z` — импортирует `X` под именем `Z`.
- **Все**: `from M import *` — импортирует все экспортируемые модулем имена (в соответствии с поведением рантайма).

Можно комбинировать: `from M import a, b as c, *`.

## Как ищутся модули (базовый путь)

При запуске скрипта из **командной строки** (например, `datacode path/to/script.dc`) **базовый путь** устанавливается в **директорию, в которой лежит этот скрипт**. Все операторы `import` и `from` разрешаются относительно этого пути (и при необходимости [путей пакетов DPM](#пути-пакетов-dpm)).

- Запуск `datacode examples/ru/15-модули/main.dc` → базовый путь = `examples/ru/15-модули/`.
- Поэтому `from core.config import get_settings` ищет `examples/ru/15-модули/core/config/__lib__.dc`.

Если код выполняется **без файла скрипта** (REPL или `run("from foo import bar")` без базового пути), локальные модули `.dc` **не находятся**; доступны только [встроенные модули](#встроенные-модули). При использовании API библиотеки нужно передавать `base_path`, чтобы работали локальные импорты (см. [Internals](./internals/module_import_system.md)).

## Два вида локальных модулей

Локальные модули ищутся относительно **базового пути** (и путей DPM). Поддерживаются два варианта:

### 1. Модуль-файл

Файл с именем `<имя>.dc` импортируется как модуль `имя`:

- Путь: `<base_path>/<имя>.dc`
- Пример: `utils.dc` → `import utils` или `from utils import foo`

### 2. Пакет (директория с `__lib__.dc`)

**Пакет** — это директория, в которой есть файл `__lib__.dc`. Импортируется пакет по имени директории; выполняется код из `__lib__.dc`.

- Путь: `<base_path>/<имя>/__lib__.dc`
- Пример: директория `core/config/` с файлом `__lib__.dc` внутри → импорт как `core.config` (см. [Составные имена](#составные-имена-пакеты)).

Загрузчик **предпочитает** пакет файлу: если есть и `<имя>/__lib__.dc`, и `<имя>.dc`, используется пакет.

## Составные имена (пакеты)

Можно импортировать **подмодуль** составным именем. Каждый сегмент разрешается по очереди:

- `from core.config import get_settings`:
  1. Найти пакет или файл `core` в базовом пути → директория `core/` с `core/__lib__.dc` (или файл `core.dc`).
  2. В ней найти `config` → `core/config/__lib__.dc` (или `core/config.dc`).
  3. Загруженный модуль — `core.config`; его экспорты (например, `get_settings`) берутся из `core/config/__lib__.dc`.

Для **вложенных пакетов** каждый сегмент, кроме последнего, должен быть **пакетом** (директорией с `__lib__.dc`). Например, чтобы работал `core.config`, нужны:

- `core/__lib__.dc` (может быть пустым или с комментарием),
- `core/config/__lib__.dc` (фактические экспорты).

---

## Что экспортируется

Для **локального модуля .dc** (файл или `__lib__.dc`) экспортом является **глобальные переменные и функции** модуля. Всё, определённое на верхнем уровне (переменные, функции, классы), доступно при импорте.

- `from M import f` — `f` должна быть глобальной в этом модуле.
- `import M` — объект пространства имён даёт доступ ко всем этим глобалам (например, `M.f`).

## Встроенные модули

Эти модули встроены в рантайм; для них **не** нужен соответствующий файл или пакет `.dc` в базовом пути:

| Модуль         | Описание |
|----------------|----------|
| `ml`           | Машинное обучение (тензоры, графы, нейросети, датасеты и т.д.) |
| `plot`         | Изображения, окна, графики (столбчатые, линейные, круговые, тепловые карты, subplots) |
| `settings_env` | Загрузка .env, Settings, Config, Field |
| `uuid`         | Генерация UUID (v4, v7), parse, to_string, байты, v3/v5 |
| `database`     | Движок БД и DatabaseCluster |

Пример:

```datacode
from database import engine, DatabaseCluster
from uuid import v4, v7
```

Если имя **не** является встроенным модулем и локальный модуль не найден (в базовом пути или путях DPM), возникает ошибка времени выполнения, например: *"Module 'X' not found. Searched in base path and DPM packages."*

## Пути пакетов DPM

Кроме базового пути скрипта рантайм может использовать дополнительные пути поиска (например, для пакетов, установленных через **DPM**). По ним разрешаются имена модулей так же, как по базовому пути (файл `<имя>.dc` или директория `<имя>/__lib__.dc`). Способ их задания зависит от окружения (CLI, HTTP-сервер или API библиотеки).

---

## Полный пример (локальный пакет)

Структура:

```
my_app/
├── main.dc
└── core/
    ├── __lib__.dc
    └── config/
        ├── __lib__.dc
        ├── base.dc
        ├── config.dc
        ├── dev_config.dc
        └── prod_config.dc
```

- `main.dc`: `from core.config import get_settings, load_settings`
- `core/__lib__.dc`: может быть пустым или с кратким комментарием (чтобы `core` был пакетом).
- `core/config/__lib__.dc`: определяет `get_settings`, `load_settings` и остальные экспорты; может использовать `from base import ...`, `from config import ...` и т.д., они разрешаются относительно `core/config/`.

Запуск из корня проекта:

```bash
cargo run --bin datacode my_app/main.dc
```

Готовый пример в репозитории:

- **RU**: [examples/ru/15-модули/](../../examples/ru/15-модули/) (см. [README](../../examples/ru/15-модули/README.md))

## Кратко

| Тема | Описание |
|------|----------|
| **Синтаксис** | `import M` / `from M import X, Y as Z, *` |
| **Базовый путь** | Задаётся директорией скрипта при запуске из CLI; относительно него разрешаются локальные модули. |
| **Модуль-файл** | `<base_path>/<имя>.dc` → модуль `имя`. |
| **Пакет** | `<base_path>/<имя>/__lib__.dc` → модуль `имя`; приоритет над файлом с тем же именем. |
| **Составное имя** | `core.config` — обход по сегментам (каждый сегмент = пакет или файл); последний сегмент — загружаемый модуль. |
| **Встроенные модули** | `ml`, `plot`, `settings_env`, `uuid`, `database` — файлы не требуются. |
| **Экспорты** | Для модулей .dc экспортом являются глобалы верхнего уровня (переменные, функции, классы). |

Детали на уровне байткода и VM (ModuleObject, перемаппинг функций, кэш) см. в [Система импорта модулей (Internals)](./internals/module_import_system.md).
